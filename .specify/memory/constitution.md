# QuickReply AI Constitution

## Core Principles

### I. Solo Hackathon MVP Focus & Mocking
All specifications and code must focus exclusively on the core value of Conversational Product Discovery and Action Calling (Add to Cart). Omnichannel features, CRM/ERP tracking, user profile management, review systems, and payments must be mocked completely using a Perfect Mocking Strategy to maintain rapid solo development.

### II. Architectural & Tech Stack Integrity
The application must be built using Next.js (App Router), Tailwind CSS, TypeScript, and deployed exclusively on the Vercel Edge Network. No third-party infrastructure besides the specified RAG data layer is permitted.

### III. Streamable UI & Function Calling
All chat widgets must implement standard chat layouts using `@ai-sdk/elements`. Interactive React components (Product Cards, Buy Now buttons, specification comparison grids) must be dynamically streamed into the chat response using Vercel AI SDK Core function calling (tools) and streamable UI.

### IV. Fail-Safe UX & Hybrid Data
Product data is retrieved realtime from Phong Vũ Discovery API via MCP tools. Policy/warranty/FAQ data uses RAG with PostgreSQL `pgvector` on Supabase. LLM timeouts or API errors must implement robust fallback mechanisms (e.g., 3x silent retries).

### V. Definition of Ready (DoR)
Before coding any feature, the specification file MUST include:
1. User Persona Alignment (online shopper benefit or performance metrics)
2. Technical Implementation Flow (step-by-step Next.js, AI SDK, and LLM tool data flow)
3. UI/UX Impact (layout mockup or description of React components streamed in chat)
4. Success Metrics (how it contributes to latency < 3s or conversion)

## Technical Commandment Enforcement

All implementation plans and task lists generated by SpecKit must check and enforce strict alignment with the Next.js, Vercel AI SDK, and Supabase tech stack rules. Any deviating backend architectures (e.g., Python/FastAPI, Express) are disallowed.

## Development Workflow & Definition of Done

Features must be implemented and tested incrementally on a per-user-story basis (P1 -> P2 -> P3) with corresponding Vitest and Playwright tests verifying the UI, state storage, and streamable component triggers.

## Governance

This constitution governs all specification generation, architectural planning, and code changes in the QuickReply AI codebase. Any updates require updating the version number and date.
